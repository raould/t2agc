# Implementation Plan: t2-agc

This document outlines the staged implementation plan for the `t2-agc` runtime, using `t2lang`. The goal is to build a tiny, deterministic, cooperative actor runtime inspired by the Apollo Guidance Computer.

---

## 📅 Stage 1: The Kernel (Scheduler & Tasks)
**Goal:** Establish the execution loop, task structure, and cooperative yielding.

### 1.1 Data Structures
Define the core `Task` structure (likely a Class or Object in t2lang).
*   **id**: Unique integer.
*   **priority**: `:low` | `:normal` | `:high`.
*   **status**: `:runnable` | `:waiting` | `:done` | `:crashed`.
*   **gen**: The generator execution context (the coroutine).
*   **mailbox**: Queue of messages.
*   **budget**: Integer (reduction count, initial e.g. 100).
*   **capabilities**: Set of held capabilities.

### 1.2 The Scheduler
Implement the `Scheduler` singleton.
*   **run-queues**: A map or array of queues for each priority level.
*   **task-registry**: Map of `pid -> Task`.
*   **current-task**: Pointer to the active task.

### 1.3 Primitives
*   `spawn(fn, args)`: 
    1.  Create `Task` struct.
    2.  Instantiate `fn` (must be a generator-fn) with `args`.
    3.  Add to `run-queues`.
    4.  Return `pid`.
*   `yield()`: 
    1.  Generator yields control back to Scheduler.
    2.  Scheduler moves task to back of queue.
*   `run()`: 
    1.  The main loop.
    2.  Selects highest priority non-empty queue.
    3.  Pops task -> `current-task`.
    4.  Calls local `step()` function which calls `task.gen.next()`.
    5.  Handles result (Done? Yielded? Error?).

**Internal Pseudocode (t2lang):**
```t2
(class Scheduler
  (field (run-queues) { :high [] :normal [] :low [] })
  (field (tasks) {})
  (field (pid-counter) 0)

  (method schedule (task)
    (let q (get (. this run-queues) (. task priority)))
    (array-push q task))

  (method run ()
    (while true
       (let task (this.pick-next-task))
       (if (not task) (break))
       (this.execute-slice task)))
)
```

## 📅 Stage 2: Messaging (Send & Basic Receive)
**Goal:** Enable tasks to communicate.

### 2.1 Mailbox
*   FIFO queue attached to `Task`.

### 2.2 `send`
*   Signature: `(send pid msg)`
*   Logic:
    1.  Look up `Task` by `pid`.
    2.  Push `msg` to `Task.mailbox`.
    3.  **Crucial:** If `Task.status` is `:waiting` (blocked on receive), change to `:runnable` and enqueue it.

### 2.3 `receive` (Primitive/Blocking)
*   The raw generator yield command: `(yield (make-primitive :receive))`
*   Scheduler logic when handling `:receive` yield:
    1.  Check `current-task.mailbox`.
    2.  If empty: Change status to `:waiting`. **Do not re-queue**.
    3.  If not empty: Pop message, feed back into generator via `.next(msg)`.

## 📅 Stage 3: Selective Receive
**Goal:** The heart of the actor model — filtering the mailbox.

### 3.1 The Challenge
`receive` isn't just popping the head. It's scanning for a match.

### 3.2 Implementation Strategy
Instead of the Scheduler doing the matching (which requires the Scheduler to know user patterns), the Scheduler provides a **peeking** iterator or a list snapshot to the Task, and the Task (in user-space code generated by the `receive` macro) decides what to pick.

**Revised `receive` Macro Logic:**
The `(receive ...)` macro expands to a loop:
1.  Iterate through mailbox messages.
2.  Test pattern `match` against message.
3.  If match: 
    *   Call `scheduler.consume_specific_message(msg_index)`.
    *   Execute body.
4.  If end of mailbox reached with no match:
    *   `yield :wait_for_new_message`.

**Scheduler Update:**
*   Handle `:wait_for_new_message`: Block task until *new* mail arrives.

## 📅 Stage 4: Capabilities & Effects
**Goal:** Enforce safety and "authority-based" side effects.

### 4.1 Structure
*   `Capability` object (opaque token).
*   `Task` holds a list of allowed capabilities.

### 4.2 `effect` Primitive
*   Signature: `(effect capability :operation arg1 ...)`
*   Macro/Runtime logic:
    1.  Verify `capability` is valid and held by `current-task`.
    2.  Perform side-effect (e.g., `console.log`, `fetch`, `setTimeout`).
    3.  Record in **Effect History**.

---

## 📅 Stage 5: Diagnostics & Determinism (The "AGC" Layer)
**Goal:** Observability and debugging.

### 5.1 History Ring Buffers
Add to `Task`:
*   `history-events`: Array of last N events `{ type: :send|:receive|:effect, data: ... }`.

### 5.2 Global Inspection
*   Create global `__t2agc__`.
*   Methods: `dump-task(pid)`, `system-status()`, `trace-message(msg-id)`.

### 5.3 AGC Codes
*   Define `(def-agc-code "1202" "Executive Overflow - Run queue length exceeded")`.
*   Emit these codes to stderr/logs when runtime limits are hit.

---

## 📅 Stage 6: Macros & Syntactic Sugar
**Goal:** Make it look like the examples.

### 6.1 `task` Macro
Wraps the `function*` definition and `spawn` call if it's a named task.

### 6.2 `defprotocol`
Stores metadata in a global registry for `Dlite` to use later. It does not affect runtime execution logic (unless we add runtime contract enforcement later).

---

## 📅 Stage 7: OTP Basics (t2-agc-otp)
**Goal:** Higher level reliability patterns.

*   **Supervisors**: Special tasks that trap exits.
*   **Behaviors**: `gen-server` equivalent loop macros.
*   **Registries**: `(register name pid)`.
